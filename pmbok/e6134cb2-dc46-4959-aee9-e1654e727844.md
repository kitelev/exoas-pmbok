# Lesson — RFC Phase 0 spike insufficient depth для library internals — caveats surface during impl

> **Closure:** [[d90daecb-9c96-440b-9a68-e1c2237d8da8|Closure — RFC 0a0791c1 — FocusProfile Phase 4]]
> **Category:** [[pmbok__LessonCategoryTechnical]]
> **Captured at:** 2026-05-31T23:22:00 (Asia/Almaty)

## Insight

**RFC Phase 0 empirical spikes** verify functional API surface (e.g., «requestUrl supports binary», «nanotar parseTarGzip handles GitHub tarball») но **don't deep-dive внутрь library internals** для performance / scalability / streaming claims. Empirical caveats surface **during implementation** when child Claude reads library source carefully.

**RFC 0a0791c1 example (caught by B.1 child 2026-05-31T23:22):**
- RFC v3 Phase 0 spike claim: «nanotar@0.3.0... `parseTarGzip` builtin async iteration»
- RFC v3 Architect #8: «streaming AsyncIterable interface (mobile mem budget) для TarExtractor»
- **Reality verified by B.1 child:** nanotar 0.3.0 has **NO streaming parser**. `parseTarGzip(arrayBuffer)` returns an **array** of TarFileItem; AsyncIterable wrapper is over that array, not a true stream. **Entire tarball buffered in memory** during pull + extract phases.
- **Implication для B.2 TarExtractor design:** «iterate-then-discard» pattern still valuable (release each Uint8Array reference after consumption), но cannot eliminate initial buffer. Memory peak ≈ tarball size + extracted blob retention.

Discovered too late to revise RFC v3 без scope ripple. Documented в B.2 task body Empirical Caveat section + B.10 regression tests adapted to realistic memory expectations.

## Context

This is a **TYPICAL pattern** — RFC author's Phase 0 spike validates «can it work?» (binary support, basic API call) но not «how does it work at depth?» (internal buffering, error modes, edge cases). Cost of missing this:
- Architect #8 review found «streaming» claim acceptable based on API surface
- Multi-agent review (Security #6, Migration) focused on supply chain + zip-slip, не on streaming internals
- B.2 task design assumed streaming → had to descope memory budget claim mid-execution

**Pre-emptive depth-check would have caught:**
- 2-line nanotar source code grep: `Array.prototype.map | yield*` patterns reveal whether truly streaming
- 5-min benchmark: extract 50MB tarball, measure peak process RSS, see whether bounded by entry size or total

Neither was в Phase 0 spike scope.

## Applicability

**Recommendation для future RFC Phase 0 spikes:**

For each library / dependency mentioned в Альтернативы и Решение:
1. ✅ Verify API surface exists (current standard practice)
2. ✅ Verify functional contract works для primary use case (current standard practice)
3. ⛔ **Verify performance/scalability claims via either:**
   - Library source read (2-5 min) — confirm streaming/lazy/buffered pattern
   - Micro-benchmark (5-10 min) — measure peak memory, latency under realistic load
4. ⛔ **Document caveats** в RFC body even if rejecting variants — surface trade-offs surface к downstream tasks

**Specific patterns to verify:**
- «streaming» / «lazy» / «backpressure» claims → read source for generators/iterators OR benchmark with mem profiler
- «zero-copy» / «in-place» claims → read for clone/spread patterns
- «O(n)» complexity claims → benchmark с N=10, 1000, 1M
- «mobile-ready» / «browser-safe» claims → check bundle deps for `bare-fs` / `node:fs` / `streamx` / Bare-runtime

**Anti-patterns:**
- Trust library README streaming claims без source read
- Assume async signature == streaming (B.1 caught: `parseTarGzip` is `async function` returning Array, not streaming generator)
- Skip benchmark «because library is small» — small libraries often skip streaming для bundle-size optimization
