# Lesson — Sequential atomic PRs outperform consolidated multi-task PR for multi-phase RFCs

> **Closure:** [[87e5f86d-998a-4ec8-b10e-b0cf2b196b07|Closure — Homoiconic plugin testing pilot]]
> **Category:** [[285d7a99-ae93-4234-a224-c9e168bf2547|pmbok__LessonCategoryProcess]]

## Insight

RFC v2's original Phase 0 spec envisioned a **consolidated PR** containing tasks 0.1-0.4 as commits + a CLI-RFC spec. During execution, tasks 0.1-0.3 shipped as **3 separate sequential PRs** (#3232, #3234, #3237). Each got own CI verdict, own code-review pass, own release tag. The CLI-RFC moved to a documentation asset only (Task 0.5).

**Empirical comparison:**

| Property | Consolidated single-PR | Sequential atomic PRs (delivered) |
|----------|------------------------|------------------------------------|
| Blast radius if regression | All 4 task changes revert together | Each PR independently revertible |
| CI signal granularity | 1 verdict for 4 tasks | 1 verdict per task — pinpoints which broke |
| Code-review attention | Reviewer faces 4 task contexts | Reviewer faces 1 task context |
| Release versioning | 1 version bump | 3 version bumps (v16.17.0/18.0/19.0) |
| Mid-flight blocker | All 4 stalled if 1 breaks | First 2 ship; 3rd can iterate |
| Wall-clock | Same total time | Same total time |
| Token overhead | -1 PR description boilerplate | +2 PR description writes (~1000 tokens) |

**Where consolidated wins:** atomic semantic commit (e.g. «cannot ship feature without all 4 parts») or when CI cost is per-PR-fixed and high.

## Where sequential won here

- **Task 0.2 caught GAS DI break during PR #3234 CI** — fix landed as separate commit in same PR; if consolidated, the DI break would have surfaced amidst 3 other tasks' changes and been harder to isolate.
- **Task 0.3 child crashed mid-execution** — orchestrator recovered by manually opening PR with the half-done commit. If consolidated, the partial commit would have been intermixed with task 0.1/0.2/0.4 changes — recovery much harder.
- **Each PR's release tag** anchored a callable git rev — task 0.1's `v16.17.0` was already published when task 0.2's PR opened, so reviewer could test against shipped feature.

## Anti-pattern

Consolidating tasks just to «save PR overhead» often costs more in blast-radius management than it saves. Token cost of sequential PR descriptions is ~1000 tokens × N-1 PRs = trivial vs hours of debugging a tangled multi-task revert.

**Rule of thumb:** **Default to sequential atomic PRs.** Consolidate only when:
1. Multiple changes form a single atomic semantic unit (incomplete = broken)
2. CI cost is dominant and fixed-per-PR (e.g. 30-min e2e suite)
3. Reviewer explicitly requests consolidation
