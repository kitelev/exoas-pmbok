# Lesson — Private sibling submodule CI clone blocked by GITHUB_TOKEN scope

> **Closure:** [[87e5f86d-998a-4ec8-b10e-b0cf2b196b07|Closure — Homoiconic plugin testing pilot]]
> **Category:** [[ddfef226-2e3a-4c16-8f25-13f5aabbbfb1|pmbok__LessonCategoryTechnical]]

## Insight

GitHub Actions `actions/checkout@v4` with `submodules: recursive` uses the default `GITHUB_TOKEN` for submodule clones. **`GITHUB_TOKEN` scope = current repo only** — it cannot clone OTHER private repos in the same organization, even if user has access to all of them locally.

**Symptoms:**
- `fatal: repository '...' not found` errors for each private submodule
- `fatal: clone of '...' into submodule path '...' failed`

**Three resolution paths:**

1. **Make sibling repos public.** Simplest if no privacy concern. Affects ALL siblings in the chain.
2. **PAT (Personal Access Token) with `repo` scope** stored as repo secret + workflow uses `git config --global url."https://${{ secrets.EXOCORTEX_PAT }}@github.com/".insteadOf "https://github.com/"`. Requires PAT lifecycle management.
3. **Vendor submodule content** — copy/sync periodically instead of git-submodule. Loses freshness guarantees.

## Context

Hit during Phase 3 Task 3.3 of homoiconic plugin testing pilot. The orchestrator created `exocortex-ems-commands-test` repo with 5 submodules: test-ontology (created PUBLIC), ems-commands (created PUBLIC), exo (already PUBLIC), ems-ontology (existing PRIVATE), shared-identities (existing PRIVATE). CI failed on the 2 pre-existing private siblings even after the 3 new ones were made public per user direction.

Parallel orchestrator (`StarterKitRetire`) hit the same wall on a different PR (#3236 → cycles 1-3 all flaky on private submodule clone). Confirmed systemic — not RFC-specific.

## Applicability

**Future RFC planning:** When designing a project that requires GitHub Actions + multi-repo submodule chain, surface the privacy boundary EARLY (RFC review stage). Two questions to answer in RFC:

- Q1: Which existing repos in the chain are private?
- Q2: Is the user willing to (a) make them public, (b) maintain a PAT, or (c) vendor content?

Without explicit answer, CI green AC will be at-risk by default. RFC should not assume default-public for siblings.

**Mitigation pattern when stuck:** Mark CI-green AC as `PartialAcceptance` + `re-escalation trigger` (machine-checkable condition for promotion to Met). Allows project to close cleanly while preserving the in-flight resolution path. Demonstrated на этом проекте — M4 marked AtRisk with `gh repo view ... --json visibility` check trigger.
