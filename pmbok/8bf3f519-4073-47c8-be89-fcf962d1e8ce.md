# Lesson — Advisor consultation BEFORE substantive work catches silent regressions

> **Closure:** [[87e5f86d-998a-4ec8-b10e-b0cf2b196b07|Closure — Homoiconic plugin testing pilot]]
> **Category:** [[285d7a99-ae93-4234-a224-c9e168bf2547|pmbok__LessonCategoryProcess]]

## Insight

Calling `advisor()` **before** committing to implementation pattern (vs post-implementation review) catches design-level regressions that would otherwise ship and require post-merge fix-forward.

**Empirical save:** Task 0.3 child consulted advisor before wiring `--seed`/`--frozen-clock` flags into `apply.action()` callback. Advisor flagged a **silent determinism regression**: instantiating `seededUidGenerator(seed)` inside `executeOnTarget()` loop would reset the counter to 0 on each target iteration → target #1 and target #2 generate the SAME UID. Test would pass on single-target case (most common); fail only on multi-target stdin-piped invocations. Bug would survive PR review (looks correct line-by-line) and only manifest in production replay scenarios.

**Fix from advisor:** derive `clock` + `uidGen` once at `.action()` callback top, pass as 6th param to `executeOnTarget()`, reuse across iterations. Implemented before commit → no fix-forward needed → PR #3237 CI 34/34 green on first try.

## Cost saved

Estimated: 1-2 follow-up PRs (detect bug in production, ship fix, write regression test) + lost user trust in determinism guarantee. Total ~3-5h orchestrator-time vs ~5 min for advisor call.

## Pattern

`advisor()` cost is **bounded** (single call, ~20-60s wall-clock + small token cost) vs **unbounded** cost of post-merge regression detection. The cost asymmetry is favorable — call advisor liberally before substantive design commitments.

**Spawn-nested-claude template (post 2026-05-23) updated:** include explicit «call advisor BEFORE substantive work» in child prompt's autonomy contract. Task 0.3 child was first to demonstrate this. Worth propagating to all future RFC-driven child prompts.

## Applicability

- **Before**: writing first code in a new design space (new module, new constructor signature, new CLI flag)
- **Before**: choosing between 2-3 valid approaches with non-obvious tradeoffs
- **Before**: applying a workaround pattern across multiple sites (risk of replicating mistake)

**NOT applicable** for: trivial follow-up edits, single-line bug fixes, mechanical refactors where pattern is established.
