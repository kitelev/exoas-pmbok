# Lesson — Code-reviewer round-1 catches CRIT PAT-leak vectors invisible к child Claude self-review

> **Closure:** [[d90daecb-9c96-440b-9a68-e1c2237d8da8|Closure — RFC 0a0791c1 — FocusProfile Phase 4]]
> **Category:** [[pmbok__LessonCategoryProcess]]
> **Captured at:** 2026-05-31T23:39:00 (Asia/Almaty)

## Insight

Для **security-critical PRs** (PAT handling, crypto, auth, secret storage), child Claude's self-review passes unit tests + advisor approval, но **misses critical attack vectors** что code-reviewer agent round-1 systematically catches. Auto-merge discipline для security-critical PRs (mandatory round-1 review pre-merge) **prevents prod regressions** that pass unit tests trivially.

**Empirical evidence (B.1 GitHubRestClient PR #3307):**
- Child Claude self-review: 57 tests pass, 93.51-100% coverage, advisor APPROVE round-2
- Code-reviewer agent round-1: caught **2 CRIT + 3 HIGH + 4 MEDIUM + 2 LOW = 11 findings**
- 2 CRIT both PAT-leak vectors author missed:
  1. `github_pat_` regex слишком narrow (caught только `gh[pous]_` token patterns) — broader fine-grained PAT format `github_pat_<alphanumeric>` would have leaked through redaction layer
  2. `pat` field stored as regular property → reflected в runtime debug logs / serialization. Changed к JS private field `#pat` для language-level hide
- Post round-1 hardening: 67 tests, coverage 93.91%, advisor round-2 APPROVE → merged

## Context

Without code-reviewer round-1 enforcement, both CRIT vulnerabilities would have shipped to prod via auto-merge:
- Public Sync deployment exposes broader-format PATs in user error logs → credential exfiltration risk
- `pat` field reflected in serialization → MITM access via plugin debug API surface

Compare B.3 silent child death case: child died **before** code-reviewer round invocation. Orchestrator resurrected via direct merge based on CI-only signal (34/34 SUCCESS) → 1 HIGH + 2 MEDIUM findings shipped, tracked в [Follow-up #3312](https://github.com/kitelev/exocortex/issues/3312). Code-reviewer round-1 **must be applied either pre-merge OR retroactively** for security-critical changes.

## Applicability

**Definition of security-critical PR:**
- Touches credential storage (PAT, OAuth tokens, API keys)
- Touches crypto primitives (signing, encryption, hashing)
- Touches auth flows (login, session, permissions)
- Touches input validation для adversarial inputs (URL parsing, file paths, tar entries)
- Touches log/error message construction (sanitization, redaction)

**Required discipline:**
- Pre-merge: invoke `code-reviewer` subagent (`subagent_type=code-reviewer`) before `gh pr merge --auto`
- Если child Claude дies before code-reviewer round → file follow-up Issue для retroactive review (cf. B.3 → #3312 pattern)
- Code-reviewer findings: ≥1 CRIT/HIGH → block merge; MEDIUM/LOW → fix-in-same-PR OR follow-up Issue per severity

**Anti-patterns:**
- Skip code-reviewer round-1 для security-critical PR «because unit tests + advisor approve» — empirically catches insufficient (B.1 case)
- Rely on advisor approval alone — advisor focuses on architecture/correctness, less on adversarial security
- Ship via CI-only signal для security-critical without retroactive code-reviewer audit
