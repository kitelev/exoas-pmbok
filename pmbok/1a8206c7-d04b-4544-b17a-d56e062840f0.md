## Insight

При закрытии проекта orchestrator (или ответственный agent) **обязан** провести Layer-3 functional smoke test в той среде, где feature реально работает — даже если все unit + integration + CI checks зелёные. CI green status — **необходимое, но не достаточное** условие functional correctness.

В этой сессии orchestrator выполнил три уровня верификации, **которые дали разные сигналы**:

1. **Unit tests** (Jest, 85/85 + новые 3 integration): GREEN
2. **CLI-equivalent execution** (`dyncommand exec` против real vault, target `36e54b4c-...`): GREEN — created asset с corrected frontmatter (4 copy-from-target, linkBackProperty applied, short targetClass)
3. **UI click в real Obsidian** через computer-control (initial attempt with installed plugin v15.169.3): **RED** — created asset has old buggy form (full IRI, `exo__Asset_source` fallback, ZERO copy-from-target)

Гэп между #2 и #3: CLI использовал worktree code (post-fix v15.173.2 equivalent), но Obsidian имел installed plugin v15.169.3 (старее). После manual update plugin → v15.173.2 + reload + cache rebuild → UI click finally produced corrected frontmatter (consistent with #2).

**Если бы orchestrator закрыл проект после #1-#2 (skipping #3)**, real-user feedback later would catch «production update lag» как separate defect cycle. Cost-of-multi-layer-verify < cost-of-discovered-defect-later.

## Context

User explicitly requested «Проверь также через computer-control» после того, как orchestrator показал успешный CLI verification. Это был **тестируемый момент** orchestrator independent-verifier protocol per memory `e20199f5-8252-4822-b431-9fff18aa680a`. Protocol сработал: каждая Done-claim проверена эмпирически, NOT trust status field или CI green как proof.

Без user prompt orchestrator might have closed проект на основе только CLI verification — что было бы semantically true (code shipped correct) но operationally недостаточно (plugin update lag).

## Applicability

**Affected pattern:** Any project where feature lives in a **deployable artifact** (plugin, npm package, binary) consumed by a separate runtime (Obsidian, browser, OS). Code merge ≠ user experience.

**Mitigation pattern:**

1. **Closure smoke test plan:** Step 4.5 of `/pmbok-close` skill is mandatory для AllAccepted outcome. Per AC → run ad-hoc end-to-end check в production-equivalent environment. UI domain → real UI click test. CLI domain → run + observe side-effect. Workflow domain → execute canonical task instance.
2. **Distinguish «code shipped» vs «feature accessible»:** «PR merged + release published» доказывает первое. «User clicks button и видит result» доказывает второе. Closure requires второе для AllAccepted; первое only — downgrade to PartialAcceptance или explicit `--skip-smoke` flag with rationale.
3. **Plugin/package update lag is a known operating reality:** at closure, **check installed version** (не just published version) before claiming functional correctness. If installed lags published — explicit update + re-test before close.
4. **Independent-verifier sample size:** ≥1 task per phase при cold-start (orchestrator-led project). При normal-flow project — ≥1 critical-path AC по re-verification at closure.
