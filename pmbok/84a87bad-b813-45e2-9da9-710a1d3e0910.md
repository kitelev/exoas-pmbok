# Lesson — tsyringe DI breaks on structural `options?:` param — use `.withDeterminism()` setter

> **Closure:** [[87e5f86d-998a-4ec8-b10e-b0cf2b196b07|Closure — Homoiconic plugin testing pilot]]
> **Category:** [[ddfef226-2e3a-4c16-8f25-13f5aabbbfb1|pmbok__LessonCategoryTechnical]]

## Insight

Adding a structural `options?: { ... }` trailing parameter to a `@injectable()` service constructor **silently breaks** any `container.resolve()` callsite. tsyringe (reflect-metadata based) tries to resolve all constructor params and fails with `TypeInfo not known for "Object"` because the structural type has no DI token.

**Workaround pattern (preserves DI compat + test ergonomics):**

```typescript
@injectable()
export class XyzService {
  private clock: IClock = liveClock();
  private uidGen: IUidGenerator = liveUidGenerator();

  constructor(@inject(DI_TOKENS.IVaultAdapter) vault: IVaultAdapter) {}

  withDeterminism(opts: { clock?: IClock; uidGenerator?: IUidGenerator }): this {
    if (opts.clock) this.clock = opts.clock;
    if (opts.uidGenerator) this.uidGen = opts.uidGenerator;
    return this;
  }
}

// Tests use fluent chain:
new XyzService(vault).withDeterminism({ clock: frozenClock(iso) });
```

Defaults applied as class field initializers (NOT in constructor body) — guarantees initialization order is field-decl-order, which happens before constructor-body assignments under TS `useDefineForClassFields: false` (default for ES2020).

## Context

Caught during CI shard failure on PR #3234 (Task 0.2 — IClock/IUidGenerator injection into 3 services). Initial implementation added `options?:` to all 3 service constructors. `GroundingExecutor` and `PreconditionEvaluator` worked fine (instantiated via plain `new XxxService()`), but `GenericAssetCreationService` is `container.resolve()`-d at `CommandManager.ts:57` → 4 CommandManager.errorHandling tests red → cascaded to 6 test-coverage shards.

Fix commit `6e917b45` reverted GAS to single-inject constructor + added the `withDeterminism()` setter. 11 new determinism tests verify partial-override + chaining + introspection.

## Applicability

- **Whenever** a service uses both `@injectable()` and structural DI params — pick ONE: either pure DI tokens or pure non-DI factory.
- **Cross-service consistency tradeoff:** GroundingExecutor + PreconditionEvaluator retain the structural-options pattern (never `container.resolve()`-d). Asymmetry is acceptable tech debt per code-reviewer; normalizing preemptively = scope creep.
- **DI library audit:** when introducing a new DI lib (tsyringe / typedi / inversify), document which params accept structural types vs require tokens — this lesson surfaced via empirical CI fail, would be quicker via doc lookup.
